### Роль `__init__.py`

Файл `ship_model/__init__.py` выполняет следующие архитектурные функции:
1. **Формирование публичного API** — явный список `__all__` определяет инкапсуляцию, экспортируя только необходимые классы при вызове `from ship_model import *`.
2. **Изоляция пространства имен** — скрывает внутреннюю структуру (относительные импорты) от внешнего скрипта.
3. **Версионирование** — задает атрибут `__version__ = "1.0.0"`.

## Архитектура и дизайн-решения

### ООП-модель и связи сущностей

В проекте реализована иерархическая модель, где центральный класс `Ship` выступает в роли фасадного контроллера для подсистем и экипажа.

#### Диаграмма отношений (ключевые связи)

```text
┌─────────────────────────┐
│          Ship           │         ┌─────────────────────┐
├─────────────────────────┤         │        Hull         │
│ - name: str             │◆───────>├─────────────────────┤ (Композиция)
│ - status: ShipStatus    │         │ - capacity: float   │
│ - crew: list            │         └─────────────────────┘
└─────────┬──────┬────────┘
          │      │                  ┌─────────────────────┐
          │      │                  │       Engine        │
          │      └─────────────────>├─────────────────────┤ (Композиция)
          │                         │ - power: int        │
          │                         └─────────────────────┘
          │ 1..*
          ▼                         ┌─────────────────────┐
┌─────────────────────┐             │    CommandBridge    │
│     CrewMember      │             ├─────────────────────┤ (Композиция)
├─────────────────────┤ <───────────┤ - current_course    │
│ - role: CrewRole    │ (Агрегация) └─────────────────────┘
│ - license_number    │
└─────────────────────┘
```

#### Ответственность классов (SRP)

| Класс | Ответственность |
|-------|-----------------|
| `Ship` | Паттерн "Фасад". Централизованное управление конечным автоматом, оркестрация подсистем. |
| `Hull` | Учет физической целостности и лимитов грузоподъемности (Capacity). |
| `Engine` | Управление запуском, типами привода и статусом исправности. |
| `CommandBridge` | Навигация, хранение курса и калибровка оборудования. |
| `SafetySystem` | Изолированный аудит оборудования (шлюпки, огнетушители). |
| `CrewMember` | Управление персональными данными, ролями и статусом дежурства (вахты). |

#### Композиция vs Агрегация

* **Композиция (Жесткая связь):** Подсистемы `Hull`, `Engine`, `CommandBridge` создаются строго внутри метода `build()` класса `Ship`. Они не могут существовать без корабля и уничтожаются вместе с ним.
* **Агрегация (Слабая связь):** Объекты `CrewMember` создаются независимо и передаются на борт через метод `add_crew_member()`. Моряк существует вне зависимости от корабля.

### Детальное описание модулей

#### ship.py — Класс Ship (Корабль)
Центральный компонент, реализующий логику переходов состояний и агрегирующий все операции.
- **`build(capacity, power)`**: стадия строительства, инициализация физических компонентов.
- **`perform_safety_check()`**: аудит готовности к плаванию (проверка мостика, двигателя и наличия капитана).
- **`sail()`**: выход в море при условии прохождения всех проверок.

#### subsystems.py — Подсистемы судна
Содержит реализацию физических модулей корабля.
- **`Hull.load_cargo()`**: пример обработки исключения `CapacityError` при попытке перегрузки.
- **`Engine.start()`**: блокировка запуска при неисправности (`SubsystemError`).
- **`CommandBridge.calibrate_systems()`**: обязательная процедура перед прокладкой курса.

#### crew.py — Класс CrewMember (Экипаж)
Управляет персоналом. Реализована логика вахт: методы `start_duty()` и `end_duty()` контролируют, может ли сотрудник выполнять задачи в текущий момент.

---

### Принципы SOLID

1.  **Single Responsibility (SRP):** Каждый модуль отвечает за свою часть: `crew.py` только за людей, `subsystems.py` только за "железо", `exceptions.py` только за ошибки.
2.  **Open/Closed (OCP):** Классы `ShipStatus`, `CrewRole` и `EngineType` реализованы через `Enum.auto()`. Новые типы можно добавлять без изменения существующей бизнес-логики.
3.  **Liskov Substitution (LSP):** Иерархия ошибок. Блоки `try/except` могут перехватывать базовый `ShipError`, при этом система корректно обрабатывает полиморфные объекты `NavigationError`, `CrewError` или `CapacityError`.
4.  **Interface Segregation (ISP):** Классы предоставляют только необходимые методы: `Ship` не перегружен внутренней логикой калибровки мостика, он делегирует эту задачу классу `CommandBridge`.
5.  **Dependency Inversion (DIP):** Класс `CLIApplication` (в `main.py`) взаимодействует с бизнес-логикой исключительно через публичные абстракции и методы главного класса `Ship`.

### Конечный автомат (FSM)

Жизненный цикл корабля строго регламентирован перечислениями статусов и проверяется в методах ядра. Благодаря реализации цикличного FSM, после швартовки в порту жизненный цикл судна не заканчивается. Навигационные системы сбрасываются, и после повторной калибровки и проверки корабль может снова выйти в рейс:

```text
  [IN_DESIGN] ───build()───> [UNDER_CONSTRUCTION]
                                      │
                              add_crew_member()
                                      │
                                      ▼
                                [EQUIPPING]
                                      │
                            perform_safety_check()
                                      │
                                      ▼
┌─────────────────────────> [READY_FOR_VOYAGE]
│                                     │
│                                   sail()
│                                     │
│                                     ▼
│                              [IN_NAVIGATION]
│                                     │
│                                   dock()
│                                     │
│                                     ▼
└────perform_safety_check()──── [MAINTENANCE]
     (после калибровки приборов)
```

### Обработка ошибочных ситуаций

Реализована собственная гранулированная иерархия исключений (наследуются от `Exception` -> `ShipError`):
* `ValidationError` — пустые строки, отрицательные числа.
* `ConstructionError` — попытки нарушить FSM на этапе постройки.
* `CrewError` — найм второго капитана, работа вне вахты.
* `NavigationError` — выход в море без проверок, швартовка в доке.
* `SubsystemError` — запуск сломанного двигателя, курс без калибровки.
* `CapacityError` — перевес груза.

## Тестирование

Проект покрыт unit-тестами с использованием фреймворка `pytest`. Фикстуры вынесены в `conftest.py` для переиспользования эталонных объектов.

**Запуск тестов:**
```bash
python3 -m pytest tests/ -v
```

**Статистика покрытия (~40+ тестов):**
* `test_ship.py`: Переходы состояний (FSM), проверка минимального экипажа, защита от двух капитанов, операции отплытия/швартовки.
* `test_subsystems.py`: Изолированные проверки двигателей, калибровки моста, ремонта корпуса и логики перегрузки (`CapacityError`).
* `test_crew.py`: Управление вахтами, валидация ФИО и лицензий.
* `test_exceptions.py`: Форматирование кастомных ошибок.
