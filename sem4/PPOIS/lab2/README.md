## 2. Архитектура приложения

### 2.1 Паттерн Model-View-Controller

Приложение построено на базе классического шаблона проектирования **Model-View-Controller (MVC)**. Разделение ответственности гарантирует, что бизнес-логика (Модель) полностью изолирована от графического отображения (Представления), а их координацию осуществляет Контроллер.

```text
┌─────────────┐   вызывает методы   ┌─────────────┐
│  Controller │ ──────────────────► │    Model    │
│             │ ◄────────────────── │             │
│             │   возвращает данные │             │
└──────┬──────┘                     └─────────────┘
       │ вызывает методы
       │ отображения
       ▼
┌─────────────┐   прямые вызовы    ┌─────────────┐
│    View     │ ─────────────────► │  Controller │
└─────────────┘     через ссылку   └─────────────┘
```

**Особенность реализации:** Контроллер (TournamentController) создает экземпляры Модели и Главного окна, передавая Представлению ссылку на самого себя. Это позволяет View напрямую вызывать управляющие методы Контроллера при возникновении событий интерфейса (нажатия кнопок, выбор пунктов меню).

### 2.2 Разделение ответственности

| Компонент | Файл | Ответственность |
|-----------|------|----------------|
| `TournamentModel` | model.py | Хранение записей, бизнес-логика поиска/удаления, пагинация |
| `MainWindow` | view.py | Построение и обновление всех виджетов tkinter, диалоги, колбэки событий |
| `TournamentController` | controller.py | Связывание модели и представления, управление пагинацией, обработка событий |
| `TournamentRecord` | model.py | Класс записи с методом расчёта вычисляемого поля `winner_earnings` |
| `XMLManager` | xml_handler.py | Чтение (SAX) и запись (DOM) XML-файлов |
| Точка входа | main.py | Создание `tk.Tk`, инстанцирование MVC-триады, запуск `mainloop` |

### 2.3 Взаимодействие компонентов

`main.py` создаёт `TournamentModel()`, `MainWindow(controller)`, `TournamentController()`.

В конструкторе контроллер получает ссылку на представление через `set_view()`.

Пользователь взаимодействует с виджетами View; виджеты вызывают методы контроллера через колбэки.

Контроллер получает управление, вызывает методы модели, затем вызывает `refresh_main_view()`.

`refresh_main_view()` получает данные из модели через `model.get_records_for_page()` и передаёт срез текущей страницы в `view.update_table(...)`.

## 3. Работа с данными

### 3.1 Структура записи

```python
class TournamentRecord:
    def __init__(self, tournament_name: str, date_held: str, sport_type: str, 
                 winner_name: str, prize_pool: float):
        self.tournament_name = tournament_name    # Название турнира
        self.date_held = date_held                # Дата проведения (строка "ДД.ММ.ГГГГ")
        self.sport_type = sport_type              # Вид спорта
        self.winner_name = winner_name            # Победитель
        self.prize_pool = float(prize_pool)       # Призовой фонд (число)
        self.winner_earnings = self._calculate_earnings()  # Вычисляемое поле (60% от фонда)
```

**Вычисляемое поле:** `winner_earnings` рассчитывается автоматически как 60% от `prize_pool` и не сохраняется в XML — при загрузке файла значение пересчитывается заново.

### 3.2 Валидация

Проверки выполняются в контроллере при добавлении записи (`add_record()`):
- Текстовые поля (`name`, `sport`, `winner`) — не могут быть пустыми после `strip()`;
- `prize_pool` — должно быть неотрицательным числом;
- `date_held` — хранится как строка, формат валидируется на уровне UI.

### 3.3 Формат XML

```xml
<?xml version="1.0" encoding="UTF-8"?>
<tournaments>
    <tournament>
        <name>The International 2024</name>
        <date_held>15.09.2024</date_held>
        <sport_type>Dota 2</sport_type>
        <winner_name>Team Spirit</winner_name>
        <prize_pool>3000000.0</prize_pool>
    </tournament>
    ...
</tournaments>
```

### 3.4 SAX vs DOM: выбор парсера

| Критерий | SAX (чтение) | DOM (запись) |
|----------|--------------|--------------|
| Модель работы | Потоковая, событийная | Дерево в памяти |
| Потребление памяти | O(1) — не строит дерево | O(n) — всё дерево в RAM |
| Удобство построения документа | Неудобно | Удобно: `createElement`, `appendChild` |
| Произвольный доступ к узлам | Нет | Да |
| **Обоснование выбора** | Эффективное чтение больших файлов | Удобное построение и форматирование (`writexml`) |

**SAX-обработчик** `TournamentSAXHandler` реализует три метода `ContentHandler`:
- `startElement` — инициализирует словарь текущей записи при теге `<tournament>`;
- `characters` — накапливает текстовое содержимое элемента;
- `endElement` — при закрытии `</tournament>` конструирует объект `TournamentRecord` и добавляет в модель.

**DOM-запись** в `save_to_xml` строит документ программно через `xml.dom.minidom.Document` и сохраняет с отступами через `writexml(indent=" ", addindent=" ", newl="\n")`.

## 4. Графический интерфейс и пагинация

### 4.1 Компоненты интерфейса

Главное окно (`MainWindow`, 1400×750 px) состоит из пяти зон, размещённых менеджером геометрии `pack`:

| Зона | Виджеты | Описание |
|------|---------|----------|
| Меню | `tk.Menu` | Файл / Правка / Данные / Справка |
| Панель инструментов | `ttk.Frame` + `ttk.Button`, `ttk.Separator` | Быстрый доступ к основным операциям |
| Основная таблица | `ttk.Treeview` + `ttk.Scrollbar` (вертикальный и горизонтальный) | Отображение записей текущей страницы |
| Пагинация | `ttk.Button` (<<, <, >, >>), `ttk.Label`, `ttk.Spinbox` | Навигация по страницам, выбор размера страницы |
| Строка состояния | `ttk.Label` (relief=SUNKEN) | Текущий статус операции |

Размещение виджетов в таблице использует комбинацию `pack` (для фреймов) и `grid` (внутри `main_frame` для `Treeview` и полос прокрутки). Такой подход позволяет таблице растягиваться при изменении размера окна (`grid_rowconfigure(0, weight=1)`, `grid_columnconfigure(0, weight=1)`).

Диалоги (`AddRecordDialog`, `SearchDialog`, `DeleteDialog`) — модальные окна `tk.Toplevel` с `transient(parent)` и `grab_set()`. Они центрируются относительно родительского окна через `winfo_x/y/width/height`.

### 4.2 Алгоритм пагинации

Состояние пагинации хранится в контроллере (`current_page: int`) и модели (`records_per_page: int`).

```python
total_pages  = ceil(total_records / records_per_page)  # минимум 1
current_page = clamp(current_page, 1, total_pages)
start_idx    = (current_page - 1) * records_per_page
end_idx      = start_idx + records_per_page
page_records = all_records[start_idx:end_idx]
```

**Сброс на первую страницу** происходит при:
- добавлении записи (`add_record`);
- удалении записей (`execute_delete`);
- загрузке файла (`load_file`);
- создании новой БД (очистка модели);
- генерации тестовых данных (`generate_test_data`);
- изменении количества записей на странице (`change_per_page`).

`ttk.Spinbox` с диапазоном 1–100 позволяет задать произвольное количество записей на странице. Диалог поиска имеет собственную независимую пагинацию результатов (использует `search_results` и `search_current_page`).

### 4.3 Автоматическое изменение ширины колонок

После каждого обновления таблицы вызывается `auto_resize_columns()`, который измеряет ширину каждой ячейки через `tkinter.font.Font.measure()` с коэффициентом 1.3 и устанавливает ширину колонки по максимальному значению.

## 5. Логика поиска и удаления

### 5.1 Структура условий

Критерии разбиты на три группы. **Внутри группы** условия объединяются через **OR**; **группы между собой** — через **AND**:

```text
(название_турнира_match OR дата_match)
AND
(вид_спорта_match OR победитель_match)
AND
(призовые_диапазон_match OR заработок_диапазон_match)
```

Группа участвует в проверке **только если хотя бы одно из её полей заполнено**. Пустые группы не влияют на результат — запись проходит фильтр по умолчанию.

### 5.2 Детали сопоставления

| Поле | Тип сравнения | Регистр |
|------|---------------|---------|
| `tournament_name` | подстрока (`in`) | нечувствительный |
| `date_held` | точное совпадение | — |
| `sport_type` | точное совпадение | — |
| `winner_name` | подстрока (`in`) | нечувствительный |
| `prize_pool` | диапазон `[min; max]` | — |
| `winner_earnings` | диапазон `[min; max]` | — |

### 5.3 Реализация в коде

```python
# controller.py — TournamentController._check_match()

# Группа 1: tournament_name OR date_held
if criteria.get("name") or criteria.get("date"):
    n_match = criteria.get("name", "").lower() in record.tournament_name.lower()
    d_match = criteria.get("date") == record.date_held
    match_g1 = n_match or d_match

# Группа 2: sport_type OR winner_name
if criteria.get("sport") or criteria.get("winner"):
    s_match = criteria.get("sport") == record.sport_type
    w_match = criteria.get("winner", "").lower() in record.winner_name.lower()
    match_g2 = s_match or w_match

# Группа 3: prize_pool range OR winner_earnings range
if has_prize or has_earn:
    p_match = p_min <= record.prize_pool <= p_max  # если задан диапазон призовых
    e_match = e_min <= record.winner_earnings <= e_max  # если задан диапазон заработка
    match_g3 = (has_prize and p_match) or (has_earn and e_match)

# Запись подходит, только если она проходит проверку по ВСЕМ заполненным группам
return match_g1 and match_g2 and match_g3
```

**Удаление** (`execute_delete`) делегирует проверку методу `_check_match`, затем удаляет найденные объекты из `self.records`. Возвращает количество удалённых записей.

## 6. Сценарии демонстрации системы

### Сценарий 1: Запуск приложения
1. Выполнить `python main.py` из директории `tournaments_app/`.
2. Открывается главное окно 1400×750 px с пустой таблицей.
3. Строка состояния отображает «Готов».
4. Счётчик показывает «Всего записей: 0», пагинация — «Страница 1 из 1».

### Сценарий 2: Генерация и просмотр тестовых данных
1. Нажать кнопку «Тестовые данные» на панели инструментов (или меню Данные → Генерировать тестовые данные).
2. В таблице появляются **55 записей** (киберспорт, теннис, футбол, шахматы, гольф, покер, снукер, велоспорт, американский футбол); строка состояния: «Сгенерировано 55 тестовых записей».
3. Счётчик: «Всего записей: 55», пагинация: «Страница 1 из 6» (при 10 записях на странице).
4. Нажать «>» — переход на страницу 2. Нажать «>>» — переход на последнюю страницу.
5. В поле Spinbox изменить значение с 10 на 20 — таблица перестраивается, пагинация: «Страница 1 из 3».

### Сценарий 3: Добавление записи
1. Нажать «Добавить» на панели инструментов.
2. Открывается модальный диалог «Добавить запись».
3. Заполнить поля:
   - Название турнира: `Чемпионат мира по программированию`
   - Дата: `15.05.2025`
   - Вид спорта: `Киберспорт`
   - Победитель: `Команда МФТИ`
   - Призовой фонд: `500000`
4. Нажать «OK» — диалог закрывается, запись появляется в таблице, пагинация сбрасывается на страницу 1.
5. Попытаться добавить запись с пустым названием — система выдаёт ошибку «Текстовые поля не могут быть пустыми», диалог остаётся открытым.
6. Ввести отрицательное значение призовых — ошибка «Призовые не могут быть отрицательными».

### Сценарий 4: Поиск записей
1. Нажать «Поиск» на панели инструментов.
2. Открывается немодальный диалог «Поиск записей».
3. **Поиск по виду спорта:** выбрать в выпадающем списке «Теннис», нажать «Найти».
   - *Ожидаемый результат:* все записи с `sport_type = "Теннис"`.
4. **Составной поиск:** ввести «Победитель» = `Карлос Алькарас`, «Призовые от» = `50000000`.
   - *Ожидаемый результат:* записи, где победитель содержит «Карлос Алькарас» **И** призовой фонд ≥ 50 млн (группы связаны через AND).
5. **Поиск по диапазону заработка:** ввести «Заработок победителя от» = `30000000`.
   - *Ожидаемый результат:* все записи, где `winner_earnings ≥ 30 000 000` (60% от фонда).
6. Результаты отображаются с пагинацией; счётчик «Найдено: N» обновляется после каждого поиска.

### Сценарий 5: Удаление записей
1. Нажать «Удалить» на панели инструментов.
2. Открывается модальный диалог «Удаление записей (Вариант 6)».
3. Ввести «Вид спорта» = `Шахматы`, нажать «Удалить».
4. Система удаляет все записи с `sport_type = "Шахматы"`, показывает «Удалено записей: 10».
5. Главная таблица обновляется, пагинация сбрасывается на страницу 1.
6. Попытка нажать «Удалить» с пустыми полями — система выдаёт предупреждение «Укажите хотя бы один критерий для удаления».

### Сценарий 6: Сохранение и загрузка XML
1. Меню Файл → Сохранить (или кнопка «Сохранить»).
2. Открывается стандартный диалог сохранения файла с фильтром `*.xml`.
3. Указать путь, нажать «Сохранить» — файл записывается в UTF-8 с отступами.
4. Меню Файл → Новый — таблица очищается, счётчик обнуляется.
5. Меню Файл → Открыть — выбрать сохранённый файл.
6. Данные загружаются через SAX-парсер; строка состояния: «Загружено из <путь>».
7. Таблица отображает загруженные записи, пагинация сбрасывается на страницу 1.
8. **Важно:** поле `winner_earnings` не сохраняется в XML и пересчитывается автоматически при загрузке.

## 8. Заключение

В ходе лабораторной работы разработано приложение управления данными о спортивных турнирах на Python с использованием архитектуры MVC и пассивной модели. Чёткое разделение ответственности между компонентами обеспечивает независимость бизнес-логики от деталей представления: модель не зависит от tkinter и может быть протестирована изолированно.

**Ключевые особенности реализации:**
- Применение **SAX для чтения** и **DOM для записи** XML демонстрирует обоснованный выбор инструмента под конкретную задачу: потоковая обработка для эффективного импорта и дерево в памяти для удобного форматированного экспорта.
- **Вычисляемое поле** `winner_earnings` (60% от призового фонда) рассчитывается автоматически при создании объекта и не сохраняется в XML, что обеспечивает целостность данных при загрузке.
- Реализованная логика поиска и удаления по составным критериям **Варианта 6** (OR внутри групп, AND между группами) обеспечивает гибкую фильтрацию данных при минимальной сложности алгоритма.
- Поддержка **постраничной навигации** с настраиваемым количеством записей на странице и независимой пагинацией результатов поиска повышает удобство работы с большими наборами данных.
